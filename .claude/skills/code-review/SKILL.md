# 代码审查

系统性审查 Pull Request、功能实现和代码变更。

## 核心原则

1. **先理解再审查** - 确保理解意图后再提建议
2. **具体且可操作** - 指出具体行号并提供明确建议
3. **平衡优点和改进** - 认可良好模式的同时建议改进

## 审查检查清单

### 正确性与逻辑
- 是否实现了声称的功能
- 边界情况是否处理
- 错误处理是否恰当
- 是否存在竞态条件

### 安全性
- 用户输入验证
- SQL 注入、XSS、CSRF 防护
- 密钥/凭证安全存储
- 认证授权检查

### 性能
- N+1 查询问题
- 不必要的数据库/API 调用
- 内存泄漏（事件监听器、订阅）

### 可维护性
- 变量/函数命名清晰
- 函数职责单一
- DRY 原则
- 魔法数字替换为命名常量

### 测试
- 正常路径和错误情况覆盖
- 测试确定性
- 边界情况测试

## 反馈严重程度

| 级别 | 说明 | 示例 |
|------|------|------|
| **[严重]** | 安全问题、数据丢失、功能损坏 | SQL 注入 |
| **[重要]** | 性能问题、逻辑错误 | N+1 查询 |
| [次要] | 代码异味、风格问题 | 命名不清晰 |
| [建议] | 可选改进 | 重构建议 |
| [表扬] | 值得强调的优秀模式 | 良好的抽象 |

## 审查流程

```
1. 阅读 PR 描述和关联 issue
2. 扫描文件列表 - 范围是否匹配
3. 审查关键路径（安全、数据完整性）
4. 检查测试覆盖率
5. 提供结构化反馈
```

## 审查响应模板

```markdown
## 概要
[PR 高层评估]

## 严重问题
- [阻塞性问题]

## 重要问题
- [非阻塞问题]

## 建议
- [可选改进]

## 审批状态
- [ ] 批准 - 可以合并
- [ ] 批准但有次要评论
- [ ] 要求修改
```
